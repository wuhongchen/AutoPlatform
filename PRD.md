# AutoPlatform 产品需求文档（PRD）

> 文档版本：v3.0  
> 更新日期：2026-04-11  
> 文档状态：执行版（与当前代码能力对齐）  
> 适用范围：`AutoPlatform` 单机部署版本（Flask + Vue + SQLite）

---

## 1. 产品定位与目标用户

### 1.1 产品定位
AutoPlatform 是面向公众号运营场景的「内容生产运营后台」，提供从灵感采集、AI 改写、模板排版到微信草稿发布的端到端流程，支持多账户并行运营。

### 1.2 目标用户
1. 个人号运营者：1-3 个公众号，追求稳定日更与低运营成本。  
2. 内容工作室：多账号矩阵，要求按账号隔离数据并快速切换处理。  
3. AI 辅助写作者：需要可控风格改写与参考灵感引用。  
4. 运营负责人：关注各状态数量、发布进度、失败重试入口。

### 1.3 核心目标（2026 上半年）
1. 建立“账户维度一致”的后台体验：统计、列表、改写参考、流水线触发保持同一账户上下文。  
2. 保证“采集 -> 改写 -> 发布”主链路可重复执行并可追踪状态。  
3. 提供可验收的 API 与状态机约束，减少前后端口径偏差。

## 2. 当前版本范围与真实能力边界

### 2.1 已落地能力（与当前代码一致）
1. 单页后台入口统一为 `/`，`/admin` 兼容重定向到 `/`。  
2. 全局账户切换（前端 Header）已接入，选择结果持久化到 `localStorage`。  
3. 账户维度联动已覆盖：概览统计、文章列表、灵感列表、改写参考灵感、流水线触发账号。  
4. 后端提供账户、灵感、文章、改写、发布、风格、模板、流水线触发、统计 API。  
5. 数据持久化使用 SQLite，默认路径为项目根目录下 `data/db/autoplatform.db`（已做 cwd 锚定）。  
6. 支持 8 个内置改写风格 + 自定义风格增删改启停。  
7. 支持 4 个模板：`default/minimal/tech/business`。

### 2.2 当前缺口（需纳入近期迭代）
1. 流水线页面仅支持“触发执行”，缺少任务列表、进度、失败重试、明细追踪。  
2. 管理后台账户页目前以“查看+切换”为主，完整创建/编辑/删除能力主要在 API/CLI。  
3. 列表接口默认上限 100 条，缺少分页与游标，数据量增大后存在性能风险。  
4. 状态字典存在历史兼容问题（如流水线历史状态 `❌ 失败`），需统一映射口径。  
5. 采集与状态更新链路需补齐服务层一致性（确保状态更新方法与业务调用完全对齐）。

## 3. 业务流程与关键场景

### 3.1 场景 A：账户切换（P0）
1. 用户在任意后台页面切换账户（或选择“全部账户”）。  
2. 系统保存 `selectedAccountId`；若账户不存在则自动回退“全部账户”。  
3. 所有支持账户维度的页面自动刷新数据。  

验收重点：切换后统计卡片、文章列表、灵感列表、改写参考、流水线触发账号保持一致，不允许出现“统计是 A 账户、列表是全部账户”。

### 3.2 场景 B：灵感库采集与采纳（P0）
1. 输入 URL + 账户，触发采集。  
2. 系统抓取正文并进行 AI 评分，生成理由与建议方向。  
3. 用户在“待决策”状态执行“采纳”，自动创建文章并进入后续处理链路。  

验收重点：采纳后必须生成 article，并在灵感记录中写入关联 article_id 与状态更新。

### 3.3 场景 C：文章改写（P0）
1. 从文章列表进入改写页。  
2. 选择改写风格、可选参考灵感（最多 5 条）、填写额外指令。  
3. 系统输出 `rewritten_html`，记录风格与参考信息。  

验收重点：改写失败要落失败状态与错误信息；成功后可直接进入发布。

### 3.4 场景 D：发布到微信草稿（P0）
1. 选择模板并发起发布。  
2. 系统使用账户微信配置生成草稿并返回草稿 ID。  
3. 文章状态更新为已发布，记录发布时间与模板信息。  

验收重点：账户未配置微信凭证时明确失败，不允许“假成功”。

### 3.5 场景 E：流水线一键处理（P0/P1）
1. 用户选择账户并触发流水线。  
2. 系统处理该账户下待处理文章（批量大小默认 3）。  
3. 逐篇执行改写和发布，失败写入错误。  

验收重点：当前版本先保证可触发可执行（P0）；任务可观测与重试能力为 P1。

## 5. 非功能需求

### 5.1 性能
1. `GET /api/health`：P95 < 100ms。  
2. `GET /api/stats`：P95 < 300ms（单机 SQLite，1 万条以内）。  
3. `GET /api/articles`、`GET /api/inspirations`：100 条返回 P95 < 1s。  
4. 改写和发布属于外部依赖调用，需异步处理并提供超时错误提示（AI 60s，微信 60s）。

### 5.2 可靠性
1. 所有状态迁移必须可追踪（时间戳 + 错误信息）。  
2. 外部调用失败不允许吞错，必须落库并可在界面重试。  
3. 关键配置（数据库路径）必须与项目根目录绑定，避免因 cwd 变化读错库。  
4. 批处理过程中单条失败不应阻塞整个批次。

### 5.3 安全
1. `.env` 管理密钥，禁止提交真实凭证。  
2. 管理接口默认仅本机监听（`127.0.0.1`）。  
3. API 错误信息需可诊断但不过度暴露敏感数据。  
4. 账户敏感字段展示遵循最小暴露原则（前端按需脱敏）。

### 5.4 可观测
1. 每个核心动作记录结构化日志：`action/account_id/entity_id/status/cost_ms/error`。  
2. 提供基础运行指标：成功率、失败率、平均耗时、最近错误。  
3. 后台可直接查看最近 N 条流水线失败原因（P1）。

## 6. API 与数据模型约束（含统计口径）

### 6.1 API 约束
1. 所有前端请求以 `/api` 为前缀。  
2. 账户维度查询统一使用 `account_id` query 参数；空值表示“全部账户”。  
3. 列表接口默认按 `created_at DESC` 返回，默认上限 100 条。  
4. `/admin` 必须重定向到 `/`；前端静态资源统一来自 `app/static/dist`。

### 6.2 统计口径（强约束）
1. 统计主入口：`GET /api/stats`。  
2. 口径定义：
   1. 当 `account_id` 为空：统计全账户数据。  
   2. 当 `account_id` 有值：仅统计该账户数据。  
3. 响应结构固定：
```json
{
  "articles": {"pending": 0, "rewritten": 0, "published": 0},
  "pipeline": {"🧲 待改写": 0, "✅ 已发布": 0},
  "inspiration": {"待决策": 0, "已采纳": 0}
}
```
4. 前端卡片总数计算必须使用“状态计数求和”，不允许硬编码某个账户（如 `default`）。

### 6.3 核心数据模型约束
1. `Account`：`account_id` 全局唯一，不可为空。  
2. `InspirationRecord`：必须有 `account_id`，采纳后必须写 `article_id`。  
3. `Article`：必须有 `account_id`；发布成功后必须写 `wechat_draft_id` 与 `published_at`。  
4. `PipelineRecord`：状态必须来自标准集合；历史异常值需映射归一。  
5. `StylePreset`：内置风格 `is_builtin=true`，不可删除或编辑核心字段。

## 7. 状态机定义

### 7.1 文章状态机（Article）
标准状态：`pending -> rewriting -> rewritten -> publishing -> published`，失败态：`failed`。  

| 触发动作 | 前置状态 | 目标状态 | 说明 |
|---|---|---|---|
| 灵感采纳创建文章 | - | pending | 新建文章默认待改写 |
| 开始改写 | pending/failed | rewriting | 进入 AI 改写 |
| 改写成功 | rewriting | rewritten | 写入改写内容 |
| 改写失败 | rewriting | failed | 写入错误信息 |
| 开始发布 | rewritten | publishing | 进入微信发布 |
| 发布成功 | publishing | published | 写入 draft_id/published_at |
| 发布失败 | publishing | failed | 写入错误信息 |

### 7.2 灵感状态机（Inspiration）
标准状态：`待分析 -> 待决策 -> 已采纳/已跳过 -> 已进入流水线`。  

| 触发动作 | 前置状态 | 目标状态 | 说明 |
|---|---|---|---|
| 采集成功入库 | - | 待分析 | 创建记录 |
| AI 分析完成 | 待分析/分析中 | 待决策 | 写入评分与建议 |
| 用户采纳 | 待决策 | 已采纳 | 业务确认采纳 |
| 生成文章并挂接 | 已采纳 | 已进入流水线 | 关联 article_id |
| 用户跳过 | 待决策 | 已跳过 | 不进入文章链路 |

### 7.3 流水线状态机（PipelineRecord）
标准状态集合：  
`🧲 待改写 -> ✍️ 改写中 -> 🧾 待审核 -> 🚀 待发布 -> 📤 发布中 -> ✅ 已发布`  
失败分支：`❌ 改写失败`、`❌ 发布失败`。  

兼容要求：历史库中存在非标准状态（如 `❌ 失败`）时，统计层须映射到失败大类，不阻塞页面展示。

## 8. 测试验收标准

### 8.1 端到端场景（Given-When-Then）

1. 账户切换联动  
Given 存在账户 A、B 且两者数据不同  
When 在顶部切换 A -> B  
Then 概览统计、文章列表、灵感列表在 3 秒内同时切换为 B 账户数据

2. 全部账户视图  
Given 当前选中“全部账户”  
When 打开概览页  
Then 文章总数等于 `articles` 各状态求和，且不依赖固定账户 ID

3. 灵感采集  
Given 输入有效 URL 与账户 A  
When 提交采集  
Then 新增灵感记录 `account_id=A`，状态进入待分析/待决策链路

4. 灵感采纳  
Given 某灵感处于待决策  
When 点击采纳  
Then 新增文章记录并回写灵感 `article_id`，灵感状态更新为已进入流水线

5. 文章改写成功  
Given 文章状态为 pending  
When 选择风格并提交改写  
Then 文章状态为 rewritten，`rewritten_html` 非空，记录 `rewrite_style`

6. 文章改写失败  
Given AI 服务不可用或超时  
When 提交改写  
Then 文章状态为 failed，`error_message` 非空

7. 发布成功  
Given 文章 rewritten 且账户微信配置完整  
When 发起发布  
Then 状态为 published，存在 `wechat_draft_id` 与 `published_at`

8. 发布失败  
Given 文章 rewritten 但账户缺少微信凭证  
When 发起发布  
Then 返回错误“微信配置不完整”，文章状态为 failed

9. 流水线触发前置校验  
Given 未选择账户  
When 点击运行流水线  
Then 前端阻止请求并提示先选择账户

10. 静态入口兼容  
Given 访问 `/admin`  
When 后端响应  
Then 返回 302 到 `/`，页面资源来自 `app/static/dist`

### 8.2 API 契约检查清单
1. `GET /api/stats` 支持可选 `account_id`。  
2. `GET /api/articles`、`GET /api/inspirations` 支持 `account_id` 过滤。  
3. `POST /api/pipeline/process` 透传 `account_id` 与 `batch_size`。  
4. 错误响应统一含 `error` 字段。  
5. 关键实体返回包含 `id/created_at/updated_at`。

### 8.3 非功能检查清单
1. 启动路径一致：从任意 cwd 启动均读取项目内数据库。  
2. `.env` 缺失时提供清晰提示，不静默失败。  
3. 列表数据量接近 100 条时页面可在可接受时延内渲染。  
4. 失败链路日志可定位到账号、文章/灵感 ID、错误信息。

## 9. 里程碑与风险

### 9.1 里程碑（建议）
1. M1（1 周）：口径收敛  
   1. 统一状态字典与统计映射。  
   2. 补齐采集/状态更新链路一致性。  
   3. 完成账户维度回归测试脚本。
2. M2（1-2 周）：流程补强  
   1. 流水线任务列表与进度展示。  
   2. 失败重试与审计字段落地。  
   3. 账户后台 CRUD 能力补齐。
3. M3（1 周）：稳定性与运维性  
   1. 列表分页与性能优化。  
   2. 结构化日志与基础指标看板。  
   3. 发布前验收清单自动化。

### 9.2 主要风险与应对
1. 外部依赖风险（AI/微信接口波动）  
   应对：统一超时、重试与降级错误提示，保留失败可重跑入口。
2. 数据口径风险（账户维度不一致）  
   应对：以 `/api/stats` 为唯一统计口径，前端禁用硬编码账户。
3. 历史数据兼容风险（旧状态值）  
   应对：建立状态映射层与迁移脚本，避免报表失真。
4. 配置风险（环境变量污染，如 `DEBUG=release`）  
   应对：启动前校验并输出可执行修复建议。

## 10. 出入范围说明

### 10.1 本 PRD 覆盖
1. 管理后台核心运营链路：账户切换、灵感、改写、发布、流水线触发、统计。  
2. 与当前代码一致的 API/模型约束与验收标准。  
3. 近期迭代优先级与里程碑。

### 10.2 本 PRD 暂不覆盖
1. 多用户权限体系。  
2. 云端托管与分布式部署。  
3. 多平台发布（知乎/头条等）与商业化计费。
