# Product Specification: Multi-Account WeChat Content Operating System

## 项目概述

本项目的产品目标，不再是一个“单篇公众号文章改写脚本”，而是基于微信公众号发布能力搭建一套可多账户独立运行的内容运营系统。系统围绕“抓取 -> 分析 -> 入库 -> 改写 -> 审核 -> 发布 -> 追踪”形成闭环，服务对象是需要持续运营多个公众号账号的个人创作者、内容团队和代运营团队。

产品核心定位：
1. 以微信公众号发布为最终交付出口。
2. 以飞书多维表格 + 飞书文档作为内容存储与流程协作中心。
3. 以本地数据库承载账户配置、运行索引、任务日志和链路追踪，而不是继续依赖 JSON 文件或内存态。
4. 以多账户配置隔离作为系统基础能力，而不是后补功能。
5. 以 AI 分析和 AI 改写提升内容生产效率，而不是把系统做成一个纯手工后台。

明确边界：
1. 本系统不自建内容 CMS，正文与链接仍然落在飞书。
2. 本系统不替代人工审核，审核节点必须保留。
3. 本系统优先支持公众号文章与飞书文档输入，其他来源属于扩展能力，不作为当前版本主目标。
4. 本地数据库只保存系统元数据，不保存完整正文主内容，避免与飞书形成双主存储冲突。

当前版本要解决的核心问题：
1. 多个公众号账户共用一套流程时，配置串号、表格串号、提示词串号，需要通过本地账户数据库和账户隔离规则彻底解决。
2. 灵感抓取与发布流水线断裂，导致原文、改写稿、发布记录难追踪，需要补齐链路 ID、映射关系和任务索引。
3. 改写、发布失败后缺少可视化排查入口和统一修复路径，需要将日志、失败原因、修复动作系统化沉淀。
4. 内容团队需要在 OpenClaw 内运行，也需要在本地独立运行，不能二选一，需要形成双运行模式下的一致行为。
5. 飞书多维表格字段过于工程化、展示不友好，需要重构字段命名、分组、状态展示和摘要信息。
6. 当前系统主要依赖后台表单和固定脚本入口，配置成本高、近期内容查看分散，需要增加类似 OpenClaw 机器人的对话式配置与运营助手能力，让用户能直接通过对话查询近期内容、调整系统参数并触发固定动作。
7. 现有灵感采集仍以“单条 URL 输入”为主，缺少“按账户独立扫码登录 -> 关注列表 -> 内容列表 -> 批量抓取入库”的完整运营链路，导致高频采集场景效率不足。

## 目标用户

### 用户类型 1：个人自媒体运营者
特征：
1. 管理 1-3 个公众号。
2. 追求高频更新，但不想手动搬运、改写、排版、上传。
3. 技术能力有限，希望通过后台页面和固定流程完成任务。

主要诉求：
1. 输入文章链接后快速得到可审可发的改写稿。
2. 能区分不同账号的内容方向和风格。
3. 失败时能快速知道问题出在哪一环。

### 用户类型 2：内容团队 / 代运营团队
特征：
1. 同时服务多个公众号账户。
2. 每个账户有独立的品牌调性、内容方向、发布规则。
3. 需要流程化分工：选题、改写、审核、发布。

主要诉求：
1. 多账户隔离，不能用错 AppID、表格、提示词。
2. 任务可追踪，可审计，可重试。
3. 能通过定时调度减少人工巡检。

### 用户类型 3：OpenClaw 场景下的自动化操作者
特征：
1. 更关注“技能调用”和“定时执行”。
2. 希望通过对话直接触发固定流程。
3. 要求系统既能走 OpenClaw 代理模型，也能走独立模型。

主要诉求：
1. 技能描述清晰，动作标准化。
2. 流程默认单次执行，避免常驻进程难控。
3. 模型来源、账户来源、运行入口都可配置。
4. 能通过对话查询近期灵感、近期改写、近期发布和失败记录，不必每次都切后台找。
5. 能通过对话修改低风险配置，并把结果安全写回系统配置中心。
6. 能通过对话对最近内容执行固定运营动作，例如送入写作链、标记跳过、查看原文与打开改后稿。
7. 能按不同业务账户独立完成微信公众号扫码登录、关注列表同步、文章列表拉取和批量抓取入灵感库。

## 核心功能

### 功能 1：本地账户数据库与基础设置
功能描述：
系统需要引入本地数据库，专门管理账户资料、运行配置、任务索引和追踪关系。每个账户独立配置微信公众号参数、飞书参数、业务表格、默认模型、默认角色、内容方向和公众号提示词方向。

输入：
1. 账户名称、账户 ID。
2. `WECHAT_APPID`、`WECHAT_SECRET`、`WECHAT_AUTHOR`。
3. `FEISHU_APP_ID`、`FEISHU_APP_SECRET`、`FEISHU_APP_TOKEN`、`FEISHU_ADMIN_USER_ID`。
4. `FEISHU_INSPIRATION_TABLE`、`FEISHU_PIPELINE_TABLE`、`FEISHU_PUBLISH_LOG_TABLE`。
5. `OPENCLAW_PIPELINE_MODEL`、`OPENCLAW_PIPELINE_ROLE`、`OPENCLAW_PIPELINE_BATCH_SIZE`。
6. `OPENCLAW_CONTENT_DIRECTION`、`OPENCLAW_PROMPT_DIRECTION`、`OPENCLAW_WECHAT_PROMPT_DIRECTION`。
7. 账户启用状态、创建时间、更新时间、最近运行时间。

字段解释：
1. `账户名称`：给运营和团队成员看的业务名称，用于区分不同公众号账户。
2. `账户 ID`：系统内部唯一标识，要求稳定、唯一、可读，用于数据库关联和任务归属判断。
3. `WECHAT_APPID`：微信公众号应用 ID，用于调用公众号发布相关接口。
4. `WECHAT_SECRET`：微信公众号应用密钥，与 `WECHAT_APPID` 配套使用。
5. `WECHAT_AUTHOR`：发布到公众号草稿时默认展示的作者名，属于内容展示字段，不属于鉴权字段。
6. `FEISHU_APP_ID`：飞书开放平台应用 ID，用于访问飞书接口。
7. `FEISHU_APP_SECRET`：飞书开放平台应用密钥。
8. `FEISHU_APP_TOKEN`：目标飞书多维表格应用的 `app_token`，用于定位具体业务 Bitable，不是登录 token。
9. `FEISHU_ADMIN_USER_ID`：用于文档授权、协作者添加和运营管理的飞书管理员用户 ID。
10. `FEISHU_INSPIRATION_TABLE`：灵感库表名，承载原文抓取、评分、推荐理由、原文文档等字段。
11. `FEISHU_PIPELINE_TABLE`：发布流水线表名，承载改写状态、改后文档、审核状态、发布状态等字段。
12. `FEISHU_PUBLISH_LOG_TABLE`：发布记录表名，承载最终草稿 ID、发布时间、发布结果等字段。
13. `OPENCLAW_PIPELINE_MODEL`：当前账户的默认改写模型，记录级模型配置可覆盖它。
14. `OPENCLAW_PIPELINE_ROLE`：当前账户的默认改写角色，本质上对应提示词模板类型。
15. `OPENCLAW_PIPELINE_BATCH_SIZE`：每次巡检最多处理多少条任务，表示单轮吞吐量，不表示模型并发数。
16. `OPENCLAW_CONTENT_DIRECTION`：账号整体的内容方向，用于约束选题分析和改写视角。
17. `OPENCLAW_PROMPT_DIRECTION`：通用改写方向，用于约束整体文风、结构和表达方式。
18. `OPENCLAW_WECHAT_PROMPT_DIRECTION`：公众号成稿方向，用于强化标题感、段落感、导语和移动端阅读风格。
19. `账户启用状态`：决定该账户当前能否参与抓取、改写和发布。
20. `创建时间`：账户创建时间。
21. `更新时间`：账户配置最近一次修改时间。
22. `最近运行时间`：该账户最近一次触发任意业务流程的时间。

输出：
1. 可激活的账户配置。
2. 数据库中的账户记录。
3. 账户级运行环境注入结果。
4. 后台中的当前激活账户状态。

业务规则：
1. 任一时刻只能有一个激活账户作为后台默认运行账户。
2. 所有后台动作默认使用当前激活账户执行。
3. 禁用账户不可执行抓取、改写、发布动作。
4. 账户配置必须写入本地数据库，重启后台后仍可恢复。
5. 本地数据库中的账户配置是系统级真相源，运行时再注入环境变量，不再把 JSON 文件当主配置。
6. 账户删除必须保留软删除或归档能力，不能直接把历史执行痕迹抹掉。

异常处理：
1. 账户不存在时，禁止激活与删除。
2. 最后一个账户不可删除。
3. 飞书或微信参数缺失时，允许保存，但运行时必须给出明确失败原因。
4. 数据库不可写时，后台必须阻止继续执行账户切换和运行任务。

优先级：高

### 功能 2：灵感库中心能力（入库编排与可运营视图）
功能描述：
系统将飞书灵感库定义为内容运营主数据面板，统一承接“手工 URL 直投”“RSS 采集结果”“微信公众号抓取子系统投递批次”三类输入，并完成入库编排、字段标准化、去重、状态初始化、原文文档生成与回填展示。微信公众号扫码、关注列表、文章列表拉取动作不属于本功能，统一由功能 9 负责。

输入：
1. 文章 URL。
2. 飞书 docx/wiki URL。
3. RSS 订阅源 URL（按账户管理）。
4. 来自功能 9 的“公众号文章批量投递包”（文章元数据 + 正文/摘要 + 来源标识）。
5. 手工新增到灵感库的待分析记录。
6. 入库策略（默认处理状态、去重策略、字段映射策略、每日批量上限）。

输出：
1. 灵感库记录（按账户隔离）。
2. 原文飞书文档链接。
3. 原文配图在飞书文档或多维表格中的转存结果。
4. 入库任务记录（来源、采集时间、成功数、失败数、去重数）。
5. 订阅源健康状态（最近成功时间、连续失败次数、最近错误信息）。
6. 对功能 9 的入库回执（成功条数、失败条数、失败原因、映射关系）。

业务规则：
1. 灵感库是后续 AI 评估、写作管理链、发布链路的统一入口，任何来源都必须先入灵感库，再进入下游流程。
2. 有 URL 且未分析的记录应自动进入抓取流程；手工 URL 输入默认走“立即采集”。
3. RSS 采集必须支持按账户独立配置，默认按定时任务执行，禁止跨账户混采。
4. 订阅源新增时必须做格式校验和首次连通性校验；校验失败不得进入定时任务。
5. RSS 采集默认只拉取“上次成功采集后新增”的内容，避免全量重复抓取。
6. 公众号来源数据由功能 9 按日批量投递到本功能；本功能只负责消费投递包并入库，不直接发起公众号抓取动作。
7. 去重规则必须至少包含：标准化 URL 去重 + 标题/发布时间辅助去重 + 来源 ID 去重，避免同一文章重复入库。
8. 抓取成功后必须回填标题、评分、推荐理由、建议改写方向、原文文档。
9. 抓取评分过低的内容允许直接标记为“已跳过”，不进入发布流水线。
10. 灵感库与流水线之间必须有可追溯的来源关系。
11. 原文记录写入后，必须在本地数据库同步保存账户 ID、灵感记录 ID、来源类型（manual_url/rss/wechat_mp）、原文链接、原文文档链接和创建时间，用于后续全链路追踪。
12. 定时采集任务执行后，必须回写任务统计与来源健康状态，便于后台与对话入口查看近期采集质量。
13. 每日批量入库任务需支持“成功但无新增”状态，不能误判失败。

异常处理：
1. 抓取失败时，不得静默失败，必须回写“待审”或失败说明。
2. RSS 源读取失败时，不能阻塞其他来源采集，必须按来源粒度记录失败原因。
3. 飞书文档创建失败时，仍应保留原始 URL，避免记录丢失。
4. 图片转存失败时，不影响文本主流程，但应写入日志。
5. URL 可访问但正文抽取失败时，必须回写“页面结构不支持/抽取失败”，不能仅写网络错误。
6. 功能 9 投递批次消费失败时，必须返回可重放回执并保留待补写队列，不得直接丢弃。

优先级：高

### 功能 3：AI 分析与选题评估
功能描述：
系统对抓取内容做 AI 评估，输出爆款潜力评分、推荐理由、核心洞察、建议改写方向，帮助运营者决定是否进入改写流水线。

输入：
1. 抓取后的标题、正文、图片、互动指标。
2. 账户级内容方向。

输出：
1. AI 爆款潜力评分。
2. AI 推荐理由。
3. 核心洞察。
4. 建议改写方向。

业务规则：
1. 分析结果必须写回灵感库，而不是只留在日志里。
2. 账户级内容方向会影响分析视角，但不能篡改原文事实。
3. 分析模块只负责“要不要做”和“该怎么做”，不直接替代改写。

异常处理：
1. 模型调用失败时，需要回写失败原因，不能只停留在控制台输出。
2. 分析结果字段缺失时，不得推进到自动同步阶段。

优先级：高

### 功能 4：改写流水线与状态机流转
功能描述：
通过飞书发布队列表驱动改写流程，将待处理选题推进为改写中、待审核、待发布、已发布或失败状态。改写支持账户级模型、角色与提示词方向控制。

输入：
1. 流水线记录。
2. 原文文档链接或原文内容。
3. 账户默认模型、默认角色。
4. 记录级 `改写模型`、`改写角色` 覆盖项。

输出：
1. 改写后飞书文档链接。
2. 改写标题、改写正文。
3. 更新后的流水线状态。

业务规则：
1. 状态机必须使用统一节点：
   - `🧲 待改写`
   - `✍️ 改写中`
   - `🧾 待审核`
   - `🚀 待发布`
   - `📤 发布中`
   - `✅ 已发布`
   - `❌ 改写失败`
   - `❌ 发布失败`
2. 记录级模型优先级高于账户默认模型。
3. 若已有有效 `改后文档链接`，默认跳过重复改写，除非明确强制重跑。
4. 改写结果必须保留账户风格差异，不能所有账号都写成同一种语气。
5. “写作管理链”后台页面不能只做工程状态堆叠，必须让运营一眼看懂当前任务卡在哪一环、下一步该做什么、失败时去哪修。
6. 状态展示必须带有图标语义，而不是只靠文字区分。每个节点至少需要有状态 icon、数量、节点说明和卡片动作。
7. 看板区域与右侧详情区域必须彻底隔离布局，不能出现列重叠、卡片遮挡、横向挤压后内容不可读的问题。
8. 同一条任务卡必须突出“标题 + 当前状态 + 所属账户 + 改写模型 + 改写角色 + 改后文档入口 + 失败备注 + 最近更新时间”，而不是让用户自己在小字里找信息。
9. 已发布、待审核、待发布、失败节点必须有明显视觉差异，不能全部做成同样的白卡片。
10. 页面必须支持“看板视图 + 任务详情联动”，点击任务卡后右侧详情区应展示链路信息、快捷动作和建议下一步。

页面展示要求：
1. 顶部操作区必须突出当前账户、批量执行、单篇即时处理、当前模型/角色和筛选条件，不得让操作入口与状态列抢空间。
2. 中部主区域使用状态看板，节点顺序必须严格对应状态机，且横向滚动只发生在看板内部，不得影响右侧详情区。
3. 每个状态列标题必须包含：
   - 状态 icon
   - 状态名称
   - 当前数量
   - 一句节点说明
4. 每张任务卡必须包含：
   - 标题
   - 来源账户
   - 模型与角色
   - 改后文档入口
   - 审核/发布提示
   - 失败原因或阻塞原因
   - 最近更新时间
5. 卡片中的链接和动作必须按钮化或 chip 化，不允许只放一串文字链接。
6. 右侧详情区必须突出：
   - 当前状态
   - 灵感来源
   - 流水线记录 ID / 链路 ID
   - 原文入口 / 改后文档入口
   - 可执行动作，如重试改写、重试发布、打开文档、标记已审
7. 页面必须加入轻量 icon 系统，用于表达状态、动作、风险、文档、账号、模型、审核、发布，不得只有纯文字。
8. 页面必须优先保证桌面端信息清晰，同时在较窄宽度下自动切换为“单列看板 + 下方详情”或“内部滚动”模式，避免覆盖。
9. 页面必须强调“下一步动作”，而不是只显示“当前处于什么状态”。

异常处理：
1. `改后文档链接` 字段格式不合法时，发布阶段必须阻断并回写原因。
2. 改写失败必须写入 `备注`，不能出现“失败但无原因”。
3. 状态更新失败时，至少要保留失败备注，避免黑盒。

优先级：高

### 功能 5：公众号审核与发布
功能描述：
系统将改写后的内容转换成适合公众号阅读的 HTML 模板，处理标题层级、导读区、图片、广告位、移动端样式，并发布到对应公众号草稿箱。

输入：
1. 改写后的飞书文档。
2. 公众号账户参数。
3. 封面图或封面生成配置。
4. 账户级公众号提示词方向。

输出：
1. 微信草稿 ID。
2. 发布记录。
3. 发布状态与失败原因。

业务规则：
1. 正式发布前必须存在人工审核节点，默认不能跳过。
2. 模板转换要兼顾标题感、段落感、导读区、重点区与手机端适配。
3. 广告位必须支持统一配置，且允许顶部、底部或双位置插入。
4. 图片需要优先转为微信可用素材链接，避免外链失效。

异常处理：
1. AppID 或 Secret 错误时，需要明确提示为当前账户公众号参数问题。
2. 发布失败时必须记录到发布日志表。
3. 文档 token 无法解析时，不得继续调用发布接口。

优先级：高

### 功能 6：后台运行控制、对话式配置与定时调度
功能描述：
系统需要同时提供“后台页面入口”和“对话式入口”两种操作方式。后台页面负责可视化控制、审核与诊断；对话式入口负责快速查询近期内容、草拟配置修改、触发固定流程和返回结构化结果。两种入口最终都写入同一套本地数据库与飞书业务状态，不允许出现双轨事实源。

输入：
1. 用户在后台点击的动作。
2. 用户在对话中输入的自然语言指令。
3. 当前激活账户。
4. 定时任务间隔。
5. 用户指定的时间范围、账户范围、内容范围和配置目标。
6. 用户在对话中指定的近期内容操作目标，例如“最近 3 条灵感”“最近一条失败发布”“把这条送进写作链”。

输出：
1. 可追踪的任务 ID。
2. 任务实时日志。
3. 调度状态。
4. 对话式返回结果，包括：
   - 当前账户摘要
   - 近期灵感/改写/发布摘要
   - 近期内容详情卡片
   - 配置变更草案
   - 已执行动作结果
   - 下一步建议
   - 待确认动作清单

业务规则：
1. 后台页面功能必须拆分为：
   - `运行控制`
   - `系统基础设置`
2. 对话式入口不是后台替代品，而是第二入口：
   - 查询类动作可直接返回结果
   - 低风险配置可先生成草案，再执行写入
   - 高风险动作必须要求明确确认
3. 默认推荐 `pipeline-once` 单次执行，不鼓励直接常驻守护。
4. 定时调度默认绑定激活账户执行。
5. 任务列表必须显示所属账户，便于排查串号问题。
6. 对话式入口必须支持至少以下能力：
   - 查询当前激活账户配置摘要
   - 查询近期灵感、近期待发布、近期已发布、近期失败任务
   - 查询最近内容的详情，包括标题、状态、所属账户、原文入口、改后稿入口、失败原因和最近更新时间
   - 修改账户级模型、角色、内容方向、公众号方向、批处理条数等低风险参数
   - 对最近内容执行固定动作，例如送入写作链、标记跳过、重试改写、重试发布、打开原文、打开改后文档
   - 触发灵感扫描、流水线单次巡检、单篇即时处理、全流程 Demo
7. “近期内容”默认指当前激活账户最近 7 天内的：
   - 灵感库新增记录
   - 改写流水线状态变更记录
   - 发布日志记录
   - 失败修复记录
8. 对话式入口处理“近期内容”时，只能做以下三类事情：
   - 查询与汇总
   - 触发已有固定动作
   - 修改低风险状态字段
   不得在对话中直接改写正文主内容，正文编辑仍应回到飞书文档或后台确认链路。
9. 对话修改配置时，必须明确“修改前 -> 修改后”的差异，不允许静默覆盖。
10. 任何会影响账户级配置、发布动作、删除动作或批量动作的对话请求，都必须先返回预览，再等待明确确认。
11. 对话入口写入的配置，必须与后台页面和本地数据库保持实时一致。
12. 对话入口不得绕过人工审核、发布确认和高风险删除动作。
13. 对话式入口需要支持 OpenClaw 内调用，也要支持本地独立运行时的同等能力。
14. 后台页面与对话入口必须复用同一套动作定义、参数校验和权限边界，避免“后台能做 / 对话不能做”或“对话能写 / 后台看不到”的分裂体验。

异常处理：
1. 禁用账户被选中执行时，必须立即阻断。
2. 调度线程出错时，不得导致后台整体崩溃。
3. 后台只聚合日志，不得替代核心业务状态回写。
4. 对话指令无法识别时，必须返回可执行候选动作，而不是直接失败。
5. 对话请求涉及敏感配置变更但缺少目标账户时，必须先要求明确账户上下文。
6. 对话返回的近期内容若数据不足，必须说明是“无记录”还是“读取失败”，不能混淆。
7. 配置写入失败时，必须把失败原因返回到对话结果，并阻止误报成功。
8. 对话引用“这条”“上一条”“最近那条失败记录”之类上下文不清的指代时，系统必须先回显匹配对象，再执行动作。
9. 对话请求命中了多条近期内容但动作只允许单条执行时，系统必须要求用户二次确认具体记录，不能默认猜测。

优先级：高

### 功能 7：失败修复、链路追踪与可观测性
功能描述：
系统需要对抓取失败、改写失败、发布失败建立统一诊断与修复路径，并且能够把“灵感记录 -> 流水线记录 -> 改后文档 -> 发布记录 -> 任务日志”串起来，而不是靠人工猜。

输入：
1. 失败记录。
2. 任务日志。
3. 飞书字段内容。
4. 本地数据库中的任务索引、账户索引和链路映射。

输出：
1. 失败原因备注。
2. 可重试记录。
3. 管理后台可见的诊断信息。
4. 跨表追踪链路。

业务规则：
1. 失败原因优先写回飞书字段，再展示在后台。
2. 改写失败与发布失败必须区分，不能混成一个“失败”。
3. 后台日志和飞书备注必须能互相印证。
4. 每个任务必须生成本地任务 ID，并与账户 ID、飞书记录 ID、发布结果建立关联。
5. 失败修复动作必须保留执行记录，避免“修过但没人知道谁修了什么”。

异常处理：
1. 如果失败原因回写再次失败，至少在任务日志中保留可读错误。
2. 如果字段结构不匹配，需要提示是表结构问题，而不是伪装成模型问题。
3. 如果飞书记录存在但本地索引缺失，系统需要支持补建索引，而不是直接判为无数据。

优先级：中

### 功能 8：飞书多维表格字段治理与展示优化
功能描述：
系统需要重新设计灵感库、流水线和发布记录表的字段结构与展示方式，让运营人员看得懂、查得到、修得动，而不是继续堆工程字段。

输入：
1. 灵感库表结构。
2. 流水线表结构。
3. 发布记录表结构。
4. 账户级业务配置。

输出：
1. 优化后的字段命名规范。
2. 字段分组方案。
3. 列展示优先级方案。
4. 用于后台和飞书双端展示的一致摘要字段。

业务规则：
1. 灵感库必须突出：标题、来源链接、评分、推荐理由、建议方向、处理状态、原文文档、所属账户。
2. 流水线必须突出：标题、数据流程状态、改写模型、改写角色、改后文档链接、审核状态、发布状态、失败原因、所属账户、链路 ID。
3. 发布记录必须突出：发布标题、发布账号、发布时间、草稿 ID、发布结果、来源流水线记录。
4. 工程型字段允许存在，但不能占据默认主视图。
5. 字段命名要统一，不允许同一含义在不同表里出现三种名字。
6. 需要设计至少 1 个摘要字段，让运营人员一眼看到这条记录卡在什么阶段、下一步该做什么。

异常处理：
1. 表结构升级不能破坏历史数据读取。
2. 新字段不存在时，需要提供自动补建或巡检脚本。
3. 字段类型不匹配时，要能给出迁移建议，而不是让用户自己猜。

优先级：高

### 功能 9：公号雷达舱（公众号关注与抓取子系统）
功能描述：
系统需要新增独立的“公众号关注与抓取子系统”，产品菜单名称统一为“公号雷达舱”。在多账户架构下对每个账户独立完成二维码登录、会话维护、关注列表同步、内容列表拉取和抓取调度。该子系统与灵感库解耦，不直接承载运营视图，而是按日生成“抓取批次”投递到功能 2。该能力必须同时提供后台可视化入口和 CLI 独立入口，且两者复用同一套采集动作定义与状态持久化。

输入：
1. 本地账户 ID。
2. 微信登录动作（发起扫码、状态查询、退出登录）。
3. 关注列表同步参数（关键词、分页、同步上限）。
4. 内容列表拉取参数（目标公众号、时间范围、页数、条数上限）。
5. 抓取执行参数（仅新增/重抓、限速、失败重试、是否跳过已抓取）。
6. 批次投递参数（目标账户、每日批次时间窗、入库上限、失败重试策略）。
7. CLI 命令参数与后台动作参数。

输出：
1. 账户级微信会话状态。
2. 账户级关注列表（公众号目录）。
3. 账户级内容列表（文章目录）。
4. 抓取任务结果（成功数、失败数、跳过数、待投递数）。
5. 向功能 2 投递的日批次结果（投递条数、入库成功数、入库失败数）。
6. 后台可见任务日志与 CLI 可见执行摘要。

业务规则：
1. 一个本地账户对应一个独立微信登录态，不允许跨账户复用登录会话。
2. 后台侧边栏必须新增一级功能菜单“公号雷达舱”，与“灵感库”并列，不得折叠到系统设置或临时工具区。
3. “公号雷达舱”菜单下必须至少包含：
   - 扫码登录/状态展示
   - 关注列表同步
   - 内容列表拉取
   - 批量抓取与批次投递
   - 任务日志与重试入口
4. CLI 必须提供与后台等价的独立动作，至少覆盖：
   - `login`
   - `status`
   - `sync-mp-list`
   - `pull-article-list`
   - `fetch-content`
   - `batch-fetch-to-inspiration`
5. CLI 必须可在不启动后台的情况下独立运行，满足定时任务与运维脚本场景。
6. 后台与 CLI 触发的抓取任务，必须统一写入本地任务日志表，字段结构保持一致。
7. 关注列表、内容列表与抓取结果必须按账户隔离存储，并支持按账户筛选查看。
8. 子系统默认按“每日批量任务”将文章列表抓取结果投递到功能 2，由功能 2 统一执行入库去重和字段回填。
9. 后台“一键全流程采集”允许把“扫码登录 -> 关注列表 -> 内容列表 -> 抓取 -> 批次投递”串成单任务执行，但登录态不可用时必须中断并提示重试扫码。
10. 多账户并行采集时必须有并发保护和速率限制，避免触发微信频控导致全局失效。
11. 会话状态、公众号目录、文章目录、抓取游标必须持久化到本地数据库，不得只放内存。
12. 投递给功能 2 的每条记录必须补齐来源字段：`source_type=wechat_mp`、`source_account_id`、`source_mp_id`、`source_article_id`。
13. 投递与入库必须采用“请求-回执”模式：功能 9 负责记录投递批次 ID，功能 2 返回入库回执，双方都可追踪和重放。

异常处理：
1. 二维码已过期时，必须返回新的二维码并标记旧二维码失效。
2. 登录成功但 token/cookie 未落盘时，必须标记为“会话不完整”，不允许直接开始抓取。
3. 公众号列表接口异常时，必须保留已同步的历史列表并回写错误原因。
4. 文章列表拉取中断时，必须记录中断游标，支持断点续拉。
5. 抓取正文失败时，需回写失败类型（登录失效/频控/网络/解析失败）并支持单条重试。
6. 投递到功能 2 失败时，抓取结果不得丢失，必须保留待投递队列或失败日志。

优先级：高

## 功能优先级

高优先级：
1. 本地账户数据库与基础设置
2. 灵感库中心能力（入库编排与可运营视图）
3. AI 分析与选题评估
4. 改写流水线与状态机流转
5. 公众号审核与发布
6. 飞书多维表格字段治理与展示优化
7. 公号雷达舱（公众号关注与抓取子系统）

中优先级：
1. 失败修复、链路追踪与可观测性

低优先级：
1. 更多内容来源接入（非 RSS，如行业社区 API、短视频平台、邮件源）
2. 账户级发布策略模板（发布时间、广告策略、栏目模板）
3. 跨账号效果对比分析报表

## AI增强功能

当前必须保留的 AI 能力：
1. 灵感评分与推荐理由生成。
2. 基于账户方向的改写提示词增强。
3. 基于公众号风格的长文重构与结构化排版。
4. 图片与营销素材的语义过滤。
5. 改写去重与智能跳过，减少重复模型调用。
6. 基于失败原因、状态和历史表现的修复建议生成。
7. 基于飞书字段展示内容自动生成运营摘要。
8. 基于自然语言指令生成配置修改草案与近期内容摘要。
9. 基于近期内容上下文自动推荐下一步动作，例如“送入写作链”“重试发布”“补文档链接”。
10. 基于 RSS 来源质量和历史采集结果自动推荐“保留/降频/停用”订阅源。
11. 基于微信来源采集质量（登录稳定性、抓取成功率、频控命中率）自动给出采集节奏与重试策略建议。

建议下一阶段补强的 AI 能力：
1. 账户级内容方向自动推荐。
2. 文章标题 A/B 候选生成。
3. 审核前风险扫描：营销违规词、夸大表述、平台敏感内容。
4. 发布后数据回流分析，反哺灵感评分模型。

明确不建议现在扩的方向：
1. 上来就做复杂的自研推荐系统，属于自嗨。
2. 上来就做全平台分发，当前主战场就是公众号，别摊大饼。
3. 上来自建数据库内容中心，飞书已经承担了当前版本的事实数据中心角色。

## 非功能性需求

1. 运行模式兼容：
   - 必须支持 OpenClaw 内调用。
   - 必须支持本地独立运行。
   - `run.sh` 必须保留。
   - 微信采集链路必须支持后台入口与 CLI 独立入口并行存在。

2. 数据边界：
   - 本地数据库用于账户、配置、日志、索引、链路追踪。
   - 飞书用于正文、文档、业务协作和发布状态主视图。
   - 两者不能职责混淆，更不能形成双主内容源。

3. 多账户隔离：
   - 账户间的公众号参数、飞书表格、提示词方向不得串用。
   - 任务日志必须能定位到账户维度。
   - 账户间的微信登录态（token/cookie/二维码状态）不得串用。

4. 稳定性：
   - 失败必须可诊断。
   - 状态推进必须幂等，重复执行不能造成严重脏数据。
   - 重启后账户配置和主要状态不丢失。

5. 可维护性：
   - 模型接入应保持 OpenAI 兼容优先，减少供应商耦合。
   - 新账户配置不应要求修改代码。
   - 表结构检查要支持节流，避免频繁扫表。
   - 后台动作与 CLI 命令必须复用同一套业务服务层，避免双实现漂移。

6. 可用性：
   - 后台移动端和桌面端都可读。
   - 操作分区明确，基础设置与运行动作不能混作一团。
   - 关键字段命名尽量统一，减少运营人员理解成本。
   - 对话式入口与后台入口的动作命名、状态文案、账户上下文必须保持一致。
   - 对话返回必须尽量结构化，优先输出“账户 + 时间范围 + 内容摘要 + 可执行动作”，而不是一大段散文式说明。

7. 安全性：
   - 密钥参数不写入公开日志。
   - 账户删除、切换、禁用要有明确约束。

8. 可迁移性：
   - 当前本地数据库建议采用 SQLite，后续应可迁移到 MySQL/PostgreSQL。
   - 数据表设计需要为未来多用户后台预留扩展空间。

## 技术栈建议

当前阶段建议继续沿用现有技术路线，不要为了“看起来高级”强行重构：
1. 后端：Python 3.9+。
2. 管理后台：Flask 轻量服务。
3. 对话式运营助手：基于 OpenClaw 技能入口或本地对话代理层，统一调用现有 `/api` 与运行命令，并复用账户数据库、任务索引和动作定义。
4. 本地数据库：SQLite 作为第一阶段正式配置中心与运行索引存储。
5. 内容中心：飞书多维表格 + 飞书文档。
6. 发布出口：微信公众号草稿接口。
7. 模型接入：OpenAI 兼容接口为主，兼容 OpenClaw 代理和独立模型。
8. 图片生成：方舟生图优先，即梦作为回退。
9. 微信列表采集：可选接入 wechat-demo-cli 作为登录态抓取桥接层，失败自动回退默认抓取器。

后续若要规模化，再考虑：
1. 将 SQLite 升级为可并发更强的数据库。
2. 将后台从单文件模板升级为前后端分离。
3. 将飞书字段治理脚本做成后台可视化配置。
4. 将对话式助手扩展为多用户、多角色的运营协作助手。
5. 将自然语言指令解析做成标准动作编排层，而不是把业务逻辑散落在多个 skill 和页面按钮里。
